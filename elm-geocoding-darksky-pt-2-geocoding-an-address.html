1: Defining our primary data model

At the top level for our app, we only care about an address and latitude and longitude coordinates. While the address‚Äô type will definitely be String, we can choose between a record or tuple to house our coordinates; however, each of these values must be a Float type, as coordinates come in decimal format. For no particular reason, we‚Äôre going to use a tuple.

type alias Model =
    { address : String
    , coords : Coords
    }


type alias Coords =
    ( Float, Float )

I like to keep my models/type aliases fairly clean and primed for re-use in type definitions, so I created a separate type alias, Coords, to represent ( Float, Float ).

2: Understanding Google‚Äôs geocode response data

Let‚Äôs take a look at what a geocoding request‚Äôs response data for Auckland looks like so we can understand what we‚Äôre working with.

{
  "results": [
    {
      "geometry": {
        "location": {
          "lat": -36.8484597,
          "lng": 174.7633315
        },
        // ...
      },
      // ...
    }
  ],
  "status": "OK"
}

If you‚Äôve set up your geocoding proxy, you can see these same results by running this command:

Œª curl localhost:5050/geocode/Auckland

We can see here that we get back a status string and a results list where one of the results contains a geometry object, and inside of that, we find location and finally, our quarry: lat and lng. If we were searching for this with JavaScript, we might find this data like so:

response.results.find(x => x['geometry']).geometry.location
// { lat: -36.8484597, lng: 174.7633315 }

What would happen in vanilla JavaScript if there were no results, or those object keys didn‚Äôt exist? Elm steps up to help us solve for the unexpected.

3: Modeling the geocode response data

Based on the geocoding response, let‚Äôs list out what we‚Äôre looking at:

  • a string, status
  • a list of results
  • each result has a geometry object
  • a geometry object has a location object
  • a location object has both lat and lng properties, each of which use decimal points

Since we‚Äôre going to need decode these bits of data and reuse the types a few times, let‚Äôs create type aliases for each of these concepts (prefixed with Geo):

type alias GeoModel =
    { status : String
    , results : List GeoResult
    }


type alias GeoResult =
    { geometry : GeoGeometry }


type alias GeoGeometry =
    { location : GeoLocation }


type alias GeoLocation =
    { lat : Float
    , lng : Float
    }

If you‚Äôre not sure what type alias means, read more about type aliases in An Introduction to Elm.

4: Creating JSON decoders

There are a number of ways to decode JSON in Elm, and Brian Hicks has written about this (and has a short book on decoding JSON), and so have many others, such as Thoughtbot. Today, we‚Äôre going to be working with NoRedInk‚Äôs elm-decode-pipeline.

First, we install the package into our project:

Œª elm package install NoRedInk/elm-decode-pipeline

In our Main.elm file, we can import what we‚Äôll need from Elm‚Äôs core Json-Decode module as well as the package we‚Äôve just installed.

-- Importing from elm core.
-- We know from our type aliases that all we're working
-- with right now are floats, lists and strings.
import Json.Decode exposing (float, list, string, Decoder)

-- importing from elm-decode-pipeline
import Json.Decode.Pipeline exposing (decode, required)

Now we can write our decoders!

decodeGeo : Decoder GeoModel
decodeGeo =
    decode GeoModel
        |> required "status" string
        |> required "results" (list decodeGeoResult)


decodeGeoResult : Decoder GeoResult
decodeGeoResult =
    decode GeoResult
        |> required "geometry" decodeGeoGeometry


decodeGeoGeometry : Decoder GeoGeometry
decodeGeoGeometry =
    decode GeoGeometry
        |> required "location" decodeGeoLocation


decodeGeoLocation : Decoder GeoLocation
decodeGeoLocation =
    decode GeoLocation
        |> required "lat" float
        |> required "lng" float

Here we declare that we‚Äôd like to decode the JSON string according to our type aliases, such as GeoModel, and we expect certain keys to have certain value types. In the case of status, that‚Äôs just a string; however, with results, we actually have a list of some other type of data, GeoResult, and so we create another decoder function down the line until we dig deep enough to find what we‚Äôre looking for. In short, we‚Äôre opting for functions and type-checking over deep nesting.

Why does this feel so verbose? Personally, I‚Äôm not yet comfortable using Json.Decode.at, which might look like

decodeString (at [ "results" ] (list (at [ "geometry", "location" ] (keyValuePairs float)))) jsonString

But with the former approach, we get to be very specific with exactly what we are expecting our data to be shaped like while maintaining clarity.

5: Building our view and listening for events

It‚Äôs time to add our view function. All we‚Äôre going for today is

  • a text input that will keep track of the address by responding to the onInput event
  • a form around the input that listens for the onSubmit event
  • a paragraph to display the coordinates; for example, Coords: (123, 456)

As usual, let‚Äôs download the official elm-lang/html package:

Œª elm package install elm-lang/html

Then let‚Äôs import what we need from it:

import Html exposing (Html, div, form, input, p, text)
import Html.Attributes exposing (placeholder, type_, value)
import Html.Events exposing (onInput, onSubmit)

Each import is a function that we can use to help generate HTML5 elements which Elm then works with behind the scenes.

view : Model -> Html Msg
view model =
    div []
        [ form [ onSubmit SendAddress ]
            [ input
                [ type_ "text"
                , placeholder "City"
                , value model.address
                , onInput UpdateAddress
                ]
                []
            ]
        , p [] [ text ("Coords: " ++ (toString model.coords)) ]
        ]

Our view function takes in our model and uses Elm functions to then render output. Great! But what are SendAdress and UpdateAddress? If you‚Äôre coming from JavaScript, you might think these are callbacks or higher-order functions, but they are not. They are custom message types (that we‚Äôll define momentarily) that will be used in our update function to determine what flow our application should take next.

6: Adding message types

Thus far, we know of two message types, Update and SendAddress, but how do we define them? If you look at our view function again, you‚Äôll see the return type Html Msg. The second part of this will be the type that we create, and our custom message types will be a part of that! This is something called a union type.

type Msg
    = UpdateAddress String
    | SendAddress
    | NoOp

We will be adding more to this shortly, but this is all we have come across thus far.

7: Writing our update function

Staying consistent with The Elm Architecture, we‚Äôll define our update function in order to update our data and fire off any commands that need happen. If you‚Äôre familiar with Redux, this is where the idea for a ‚Äúreducer‚Äù came from.

This is tough to do in a blog post, so please be patient, and we‚Äôll walk through this:

update : Msg -> Model -> ( Model, Cmd Msg )
update msg model =
    case msg of
        UpdateAddress text ->
            ( { model | address = text }
            , Cmd.none
            )

        SendAddress ->
            ( model, sendAddress model.address )

        -- more code here shortly...

        _ ->
            ( model, Cmd.none )

Let‚Äôs walk through this step-by-step:

  • if the message type is UpdateAddress, then
    1. we‚Äôre expecting a string (defined in our union type)
    2. we‚Äôll call the argument text
    3. we‚Äôll then return a tuple of our updated model and a Cmd to essentially do nothing else (it‚Äôll pass through the union type and settle on the NoOp)
  • if the message type is SendAddress, then
    1. we‚Äôll accept no parameters
    2. we‚Äôll return a tuple of our model with no changes and a command that we haven‚Äôt defined yet. This is where we call the function that will actually go and get our geocode data!

8: Making our request

In order to build and send HTTP requests, we‚Äôll need to make sure we download the elm-lang/http package:

Œª elm package install elm-lang/http

and import it:

import Http

In our update function, we referenced a function named sendAddress and passed it our model‚Äôs address as a parameter. This function should accept a string, initiate our HTTP request and return a command with a message.

sendAddress : String -> Cmd Msg
sendAddress address =
    Http.get (geocodingUrl address) decodeGeo
        |> Http.send ReceiveGeocoding

geocodingUrl : String -> String
geocodingUrl address =
    "http://localhost:5050/geocode/" ++ address

Our sendAddress function does this:

  1. it builds a GET request using two arguments: a URL (derived from geocodingUrl) and our decodeGeo decoder function
  2. it then pipes the return value from Http.get to be the second argument for Http.send

Note that Http.send‚Äôs first argument is a Msg that we haven‚Äôt defined yet, so let‚Äôs add that to our Msg union type:

type Msg
    = UpdateAddress String
    | SendAddress
    | ReceiveGeocoding (Result Http.Error GeoModel)
    | NoOp

Basically, we‚Äôll either get back an HTTP error or a data structure in the shape of our GeoModel.

9: Handling the geocode response

Finally, we now need to handle the successful and erroneous responses in our update function:

update : Msg -> Model -> ( Model, Cmd Msg )
update msg model =
    case msg of
        UpdateAddress text ->
            ( { model | address = text }
            , Cmd.none
            )

        SendAddress ->
            ( model, sendAddress model.address )

        ReceiveGeocoding (Ok { results, status }) ->
            let
                result =
                    case status of
                        "OK" ->
                            results
                                |> List.head
                                |> Maybe.withDefault initialGeoResult

                        _ ->
                            initialGeoResult

                location =
                    result.geometry.location

                newModel =
                    { model | coords = ( location.lat, location.lng ) }
            in
                ( newModel, Cmd.none )

        ReceiveGeocoding (Err _) ->
            ( model, Cmd.none )

        _ ->
            ( model, Cmd.none )


-- This should go with other `init`s
-- but is placed here for relevance
initialGeoResult : GeoResult
initialGeoResult =
    { geometry =
        { location =
            { lat = 0
            , lng = 0
            }
        }
    }

Instead of having success/error logic inside one ReceiveGeocoding case match, we use Elm‚Äôs pattern matching to allow us to match on the message and Ok or Err results.

Again, let‚Äôs do this step-by-step:

  • ReceiveGeocoding is OK
    1. we destructure the response into results and status variables
    2. we check the value of status from the response to make sure all is well
    3. if status is "OK", we try to get the first item in the results list and fallback to initialGeoResult if there are no results (I love Elm for enforcing this)
    4. if status is not "OK", we fall back to the initialGeoResult
    5. we then access the location record, build an updated model record, and return it
  • ReceiveGeocoding is Err
    1. we simply return the model

10: Final wiring up with the main function & defaults

Now that we‚Äôre through the core of the application‚Äôs contents, we can wire up the remaining bits and get it to compile:

-- Define our HTML program
main : Program Never Model Msg
main =
    Html.program
        { init = init
        , view = view
        , update = update
        , subscriptions = subscriptions
        }


-- Here is our initial model
init : ( Model, Cmd Msg )
init =
    ( initialModel, Cmd.none )


initialModel : Model
initialModel =
    { address = ""
    , coords = ( 0, 0 )
    }


-- We're not using any subscriptions,
-- so we'll define none

    Sub.none
